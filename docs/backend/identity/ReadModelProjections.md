## Overview
The Read Model Projections module is responsible for maintaining denormalized, query-optimized summary tables that serve the Administration Dashboard and other potential consumers. It subscribes to domain events from the User Management, Merchant Management, and Card Provisioning modules (as well as cross-context events like `AccountCreatedEvent` from Ledger) and materializes the data into the `user_summaries` and `merchant_summaries` tables. This module fully decouples the write schema from the read schema, enabling independent evolution and future scaling.

---

## Business Rules

### 1. Specification (WHAT)
*   **Materialization Triggers:** The module handles the following events:
    *   `UserRegisteredEvent` – inserts a baseline row into `user_summaries`.
    *   `AccountCreatedEvent` – updates the corresponding summary with `account_id`, `account_number`, `currency_code`, and initializes `balance` to `'0.00'`.
    *   `CardAssignedEvent` – updates the summary with the assigned `card_number`.
    *   `MerchantOnboardedEvent` – inserts/updates a row in `merchant_summaries`.
    *   `MerchantActivatedEvent` / `MerchantDeactivatedEvent` – updates the `is_active` status in the merchant summary.
*   **Monetary Storage:** Balances in `user_summaries` are stored as `TEXT` (e.g., `'0.00'`). This preserves exact decimal representation generated by the Domain’s `Money` VO.
*   **Summary Table Schemas:**
    ```sql
    CREATE TABLE IF NOT EXISTS user_summaries (
        user_id INTEGER PRIMARY KEY,
        name TEXT NOT NULL,
        phone_email TEXT NOT NULL,
        account_id INTEGER,
        account_number TEXT,
        card_number TEXT,
        balance TEXT DEFAULT '0.00',
        currency_code TEXT
    );
    CREATE TABLE IF NOT EXISTS merchant_summaries (
        id INTEGER PRIMARY KEY,
        name TEXT NOT NULL,
        api_key TEXT NOT NULL,
        is_active BOOLEAN NOT NULL DEFAULT 1
    );
    ```

### 2. Rationale & Trade-offs (WHY)
*   **Why Materialized Read Models instead of SQLite VIEWs?** Event-driven materialization physically decouples the read schema from the write schema. This allows the read models to evolve independently and prepares the system for a future migration where the event bus is replaced by a message broker (e.g., Kafka) populating a separate Read Database (e.g., MongoDB/Elasticsearch).
*   **Why `TEXT` for Balances?** SQLite’s `REAL` type is floating-point and prone to precision loss (e.g., `0.1 + 0.2 = 0.30000000000000004`). Storing as `TEXT` preserves the exact string representation generated by the Domain’s `Decimal`-based `Money` Value Object, ensuring UI accuracy without complex rounding logic at the query layer.

### 3. Rejected Alternatives
*   **Rejected:** *Using SQLite `VIEW`s for `user_summaries` and `merchant_summaries`.*
    *   **Reason:** VIEWs tightly couple the query interface to the normalized write schema. If the `Ledger` context changes its table structure, the Identity read models would break. Materialization ensures Bounded Context isolation.
*   **Rejected:** *Performing aggregation on the fly in the dashboard.*
    *   **Reason:** Would lead to N+1 queries and complex join logic in the controller, violating separation of concerns.

## Backend Architecture

### 1. Specification (WHAT)
*   **Application Layer – Event Handlers:**
    *   `UserRegisteredReadModelHandler` – subscribes to `UserRegisteredEvent`, inserts a new row into `user_summaries`.
    *   `AccountCreatedReadModelHandler` – subscribes to `AccountCreatedEvent`, updates the `user_summaries` row matching the owner (user or merchant) with account details and sets balance to `'0.00'`.
    *   `CardAssignedReadModelHandler` – subscribes to `CardAssignedEvent`, updates the `user_summaries` row with the `card_number`.
    *   `MerchantOnboardedReadModelHandler` – subscribes to `MerchantOnboardedEvent`, inserts into `merchant_summaries`.
    *   `MerchantStatusReadModelHandler` – subscribes to `MerchantActivatedEvent`/`MerchantDeactivatedEvent`, toggles `is_active`.
*   **Database Interaction:** Handlers use the same Unit of Work (and its underlying SQLite connection) to write to the summary tables. The summary tables are defined in isolated schema files (e.g., `merchant_summaries_schema.py`).
*   **Idempotency:** Handlers are designed with idempotent SQL (e.g., `INSERT OR REPLACE`, `UPDATE` with exact matching). In the in-memory synchronous bus, replay is not an issue; the design prepares for future async delivery.

### 2. Rationale & Trade-offs (WHY)
*   **Why separate handler per event?** Each handler has a single responsibility, making it easy to test and modify independently. If a new event needs to update the summary, a new handler is added without touching existing code.
*   **Why use the same UoW?** Ensures that the read model update is part of the same transaction as the write operation (when events are dispatched synchronously). This maintains strong consistency; a failure in a read model handler will roll back the whole request.

### 3. Rejected Alternatives
*   **Rejected:** *A single monolithic handler that processes all events.*
    *   **Reason:** Would become a God object, hard to maintain and test.

## API Contract / Integration

### 1. Specification (WHAT)
*   **Contract Type:** This module has no public API. It is a passive consumer of domain events.
*   **Subscribed Events:**
    *   `UserRegisteredEvent`
    *   `AccountCreatedEvent` (from Ledger)
    *   `CardAssignedEvent`
    *   `MerchantOnboardedEvent`
    *   `MerchantActivatedEvent`
    *   `MerchantDeactivatedEvent`
*   **Data Provided to Dashboard:** The `user_summaries` and `merchant_summaries` tables are directly queried by the dashboard (via a read-model repository or raw SQL) to render pages.

### 2. Rationale & Trade-offs (WHY)
*   **Why no REST endpoint to query summaries?** The dashboard queries the tables directly within the same process for simplicity. A future microservices split would wrap these in an API, but the current in-process access is optimal for MVP performance.

### 3. Rejected Alternatives
*   **Rejected:** *Exposing a GraphQL query layer for summaries.*
    *   **Reason:** Introduces an additional layer of complexity; the dashboard’s needs are simple, and direct SQL queries are sufficient.

## Execution Flows

### 1. Specification (WHAT)
**Flow: Materializing a Complete User Record**
1.  `UserRegisteredEvent` fired -> handler inserts baseline row (`user_id`, `name`, `phone_email`).
2.  Later, Ledger creates an account -> `AccountCreatedEvent` -> handler updates the row with `account_id`, `account_number`, `currency_code`, `balance = '0.00'`.
3.  Card Provisioning emits `CardAssignedEvent` -> handler updates the same row with `card_number`.
4.  The `user_summaries` row is now fully populated for the dashboard.

**Flow: Merchant Summary Maintenance**
1.  `MerchantOnboardedEvent` -> handler inserts row with `id`, `name`, `api_key`, `is_active = 1`.
2.  On toggle, `MerchantActivatedEvent`/`MerchantDeactivatedEvent` -> handler sets `is_active` accordingly.

### 2. Rationale & Trade-offs (WHY)
*   **Why incremental updates instead of rebuilding the entire row on each event?** Building the full row would require the handler to fetch data from multiple write tables, increasing coupling. Incremental updates keep each handler minimal.
*   **Why synchronous updates?** The in-memory event bus executes handlers sequentially. This guarantees that when the HTTP request finishes and the UI reloads, the read models are 100% consistent with the write models. This eliminates “flash of stale content” in the admin dashboard.

### 3. Rejected Alternatives
*   **Rejected:** *Using a database trigger to update summaries.*
    *   **Reason:** Triggers hide logic and are hard to test. Event handlers provide explicit, traceable flows.

## Edge Cases & Known Issues

### 1. Specification (WHAT)
*   **Orphaned Read Models:** If an event handler fails *after* the UoW commits but *before* the read model updates, the UI will show stale data. With the synchronous bus, this failure crashes the request and triggers a UoW rollback, preventing orphans.
*   **Concurrent Updates:** SQLite’s single-writer lock serializes writes, so two simultaneous events targeting the same summary row are processed sequentially without data corruption.
*   **Missing Intermediate State:** If a handler fails and the transaction rolls back, no partial data is written. Retry of the same command will replay all events correctly.

### 2. Rationale & Trade-offs (WHY)
*   **Why accept the risk of eventual consistency in a future async broker?** The current architecture avoids it. When migrating to an async broker, a Saga or Outbox pattern will be required. The current design acknowledges this as a known future refactoring point.
*   **Why no explicit idempotency keys in the MVP?** Because the bus is synchronous and each event is processed exactly once per command. Idempotency will be added when moving to at-least-once delivery.

### 3. Rejected Alternatives
*   **Rejected:** *Using a background job queue for read model updates.*
    *   **Reason:** Adds operational complexity (Celery, Redis) and introduces eventual consistency, which would complicate the dashboard UI expectations in the MVP.

## Architectural Notes & Rejected Decisions

### 1. Current Implementation (WHAT)
*   **Schema Aggregation:** Summary table schemas are defined in isolated files and imported into the master database initializer, maintaining modularity.
*   **DI Registration:** Each event handler is registered with the event bus via the DI container, allowing easy addition of new handlers.

### 2. Discarded Alternatives & Reasons (WHY)
*   **Rejected:** *Materializing summaries using application-layer queries instead of dedicated event handlers.*
    *   **Reason:** Breaks the event-driven architecture and couples the read side to the write side.
*   **Rejected:** *Storing summaries in a NoSQL database from the start.*
    *   **Reason:** MVP simplicity favours SQLite; the event handler abstraction makes the swap to a different store a mere infrastructure change later.
