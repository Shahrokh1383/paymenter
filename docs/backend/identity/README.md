## 1. Purpose & Scope

The **Identity** bounded context is the systemвҖҷs authoritative source for all **actors** and their **authentication materials**.  
It manages:

- **Users** вҖ” Natural persons who interact with the platform and hold financial accounts.
- **Merchants** вҖ” Business entities that process payments, identified by a unique API key, and linked to a settlement account.
- **API Key** generation and lifecycle (for Merchants).
- The concept of a **card mapping** for Users (`user_cards`).

**Out of scope** (responsibility of other contexts):  
- Account balances, transactions, and currency lifecycle вҶ’ **Ledger**  
- Payment sessions, OTP verification, gateway tokenization вҶ’ **Checkout**  
- Sending emails, SMS, webhooks вҶ’ **Notifications**

This document is a **highвҖ‘level, stable reference** that will not change with implementation refactorings.  
For implementation details, current code locations, and known technical debt, refer to the detailed module documents listed in [Section 8](#8-module-index).

## 2. Context Boundaries

| Context | Owns & Manages | IdentityвҖҷs Interaction |
|---------|----------------|------------------------|
| **Identity** | Users, Merchants, API Keys, UserвҖ‘Card mappings | вҖ” |
| **Ledger** | Accounts, balances, transactions, currencies | Identity requests account creation (via event) and consumes currency/balance events for read models |
| **Checkout** | Payment orchestration, OTP, tokenization | Authenticates Merchants via API Key (provided by Identity) |
| **Notifications** | Dispatch of messages | Listens to events emitted by Identity (e.g., `UserRegistered`) |

**Rule:** CrossвҖ‘context communication occurs exclusively through **domain events** (final state) or a temporary AntiвҖ‘Corruption Layer (current transitional state). Direct database access across contexts is never permitted in the target architecture.

## 3. Domain Model

### 3.1 Aggregates

#### User
- **Identity:** `phone_email` (unique, composite identifier).
- **Lifecycle:** Created in a transient state (`id=0`) and assigned a permanent `id` by the persistence layer.
- **Relationships:**  
  - Holds 0..* accounts in Ledger (created after registration).  
  - Has 0..* cards mapped via `user_cards`.

#### Merchant
- **Identity:** `api_key` (unique, systemвҖ‘generated, immutable).
- **State:** `is_active` (toggled via `toggle()` method вҖ” soft delete).
- **Settlement:** Must be linked to a valid `settlement_account_id` (an account in Ledger).
- **Creation:** Onboarding automatically provisions a settlement account.
- **Invariant:** At least one active currency must exist before a merchant can be onboarded.

#### Currency (ReadвҖ‘Model)
- Identity **does not own** the Currency aggregate; it only maintains a local readвҖ‘model for reference.  
- The authoritative source is Ledger; Identity updates its view by consuming `CurrencyActivated`/`CurrencyDeactivated` events.

### 3.2 Value Objects

- **`ApiKey`** вҖ“ Immutable, format `pay_{32 bytes of urlsafe base64}`. Generated by the domain, never supplied externally.  
- **`PhoneEmail`** *(future)* вҖ“ Validated composite contact identifier. Currently a raw string (see Module 1).  
- **`CurrencyCode`** вҖ“ ISO 4217 code (e.g., `USD`). Used for referencing currencies without owning them.

### 3.3 Domain Services & Ports

| Port | Purpose | Current Implementation | Target Implementation |
|------|---------|------------------------|-----------------------|
| `AccountProvisioningPort` | Create a default Ledger account for a user/merchant | Direct SQL adapter (technical debt) | **Removed** вҖ” replaced by `UserRegistered` / `MerchantOnboarded` events |
| `UserRepository` | Persist and query Users | SQLite (with crossвҖ‘context JOINs) | CQRS: local user summary read model |
| `MerchantRepository` | Persist and query Merchants | SQLite (with crossвҖ‘context JOINs) | CQRS: local merchant summary read model |

> **Note:** The abstract ports above define the **contracts** for persistence. Their concrete implementations are pluggable and not part of the core domain.

## 4. Business Invariants & Rules

1. **User**  
   - `phone_email` must be unique across all users.  
   - A default account in a valid currency must be provisioned immediately after registration (eventual consistency allowed after eventвҖ‘driven transition).  

2. **Merchant**  
   - `api_key` is immutable and globally unique.  
   - `settlement_account_id` must reference an existing Ledger account.  
   - Toggle operation (`is_active`) does not delete any data; it only marks the merchant as inactive.  

3. **CrossвҖ‘Aggregate**  
   - A merchant cannot be onboarded if no active currency exists.  
   - All currency operations (creation, activation/deactivation) are performed by the Ledger context. Identity must never modify the `currencies` table directly.

4. **API Key**  
   - No client may set or change an API key; it is generated by the domain during merchant creation.  
   - Must conform to format `pay_{token_urlsafe(32)}` and pass validation (future).

## 5. Integration with Other Contexts

### 5.1 Target Architecture (EventвҖ‘Driven)

This is the **final, stable design** that the system evolves toward.

```
Identity Context                         Ledger Context
     |                                        |
     |  UserRegisteredEvent                   |
     |  (user_id, currency_code)              |
     |--------------------------------------->|  --> Creates default account
     |                                        |
     |  MerchantOnboardedEvent                |
     |  (merchant_id, settlement_currency)    |
     |--------------------------------------->|  --> Creates settlement account
     |                                        |
     |  AccountProvisionedEvent               |
     |  (account_id, owner_type, owner_id)    |
     |<---------------------------------------|  --> Identity updates local read models
     |                                        |
     |  CurrencyActivatedEvent /              |
     |  CurrencyDeactivatedEvent              |
     |<---------------------------------------|  --> Identity updates local currency cache
```

- **No direct database access** between contexts.  
- Each context maintains its own **read models** (CQRS) by consuming events.  
- The `AccountProvisioningPort` and its SQL adapter are deleted.

### 5.2 Current Transitional State

The system currently uses a **temporary AntiвҖ‘Corruption Layer** (`LedgerAccountProvisioningAdapter`) that performs direct SQL inserts into Ledger tables.  
This is a known violation of the bounded context isolation and will be replaced stepвҖ‘byвҖ‘step.  
**New features must not extend this adapter; they must follow the eventвҖ‘driven path.**

## 7. Design Principles & Decisions

1. **AggregateвҖ‘Based State Change** вҖ“ All modifications go through aggregate roots (`User`, `Merchant`) that enforce invariants.
2. **CQRS Read Models** вҖ“ Queries are served by dedicated, denormalised read models updated via domain events. This eliminates crossвҖ‘context JOINs.
3. **Ports & Adapters** вҖ“ The domain defines ports; infrastructure provides adapters. The domain never depends on infrastructure details.
4. **EventвҖ‘Driven Integration** вҖ“ All crossвҖ‘context communication uses domain events, ensuring loose coupling and independent deployability.
5. **API Key as a Value Object** вҖ“ Encapsulates generation and validation, preventing accidental leakage or mutation.
6. **Settlement Account as a Reference** вҖ“ Merchants reference a Ledger account by ID only; they do not duplicate account data.

## 9. Roadmap to Target State

The current codebase contains known technical debt that deviates from the target architecture.  
The following steps will bring Identity to the **stable design** described in this document:

1. **Eliminate direct SQL writes** вҶ’ Replace `LedgerAccountProvisioningAdapter` with event emission.
2. **Build local read models** вҶ’ Remove JOINs with Ledger tables; consume `AccountProvisioned` and `Currency*` events.
3. **Remove synthetic users for merchants** вҶ’ Use polymorphic owner references in Ledger.
4. **Extract Identity DI module** вҶ’ Register handlers in central DI container.
5. **Drop deprecated currency commands** вҶ’ All currency operations delegated to Ledger.

Once these steps are completed, this highвҖ‘level architecture document will exactly mirror the codebase, and no further changes will be required to it.
